# Nest Admin 系统管理模块产品文档

## 1. 文档信息

- 文档名称：Nest Admin 系统管理模块产品文档
- 文档版本：v1.0
- 适用对象：产品、后端、前端、测试
- 输出目的：基于当前 `nest-admin` 代码现状，明确系统管理模块的产品范围、功能边界、数据模型、API 规划与分期路线，作为后续迭代的统一依据。

---

## 2. 背景与目标

### 2.1 当前项目现状

当前 `nest-admin` 已具备一套**轻量后台骨架**，并已经落地以下能力：

1. **认证中心**
   - 注册
   - 登录
   - Token 刷新
   - 登出
   - 个人信息获取
   - 修改密码

2. **用户管理基础能力**
   - 分页查询用户
   - 创建用户
   - 查看单个用户
   - 编辑用户
   - 软删除用户

3. **角色管理基础能力**
   - 查询角色选项
   - 查询指定用户角色
   - 修改指定用户角色

4. **基础权限控制**
   - JWT 鉴权
   - 角色守卫
   - 当前角色枚举为：`admin` / `user`
   - 管理接口默认要求 `admin`

5. **接口文档与统一返回结构**
   - Swagger 文档
   - 全局统一响应结构：`{ code, message, data }`
   - 全局统一异常结构：`{ code, timestamp, path, message }`

### 2.2 当前痛点

虽然骨架已具备，但从“系统管理模块”的完整产品视角看，还存在明显缺口：

1. **角色模型过轻**
   - 当前仅支持 `admin/user` 二元角色
   - 尚未支持“角色实体化、权限点绑定、角色数据范围”

2. **系统管理模块不完整**
   - 缺少菜单管理
   - 缺少部门/岗位/组织架构
   - 缺少字典管理
   - 缺少系统参数配置
   - 缺少登录日志 / 操作日志 / 审计能力

3. **用户模型偏基础**
   - 缺少状态字段（启用/禁用）
   - 缺少最后登录时间、最后登录 IP
   - 缺少备注、创建人、更新人等审计属性

4. **前后端协同边界尚未产品化**
   - 接口虽然可用，但缺少“页面级产品定义”和“模块级业务规则文档”
   - 不利于前端管理台后续系统化建设

### 2.3 本次产品目标

本次要输出的是一份**可执行的系统管理模块产品文档**，目标是：

1. 让 `nest-admin` 的系统管理能力从“接口骨架”升级为“完整产品模块规划”
2. 保持当前项目的**轻量、可维护、易扩展**风格
3. 为下一步开发提供：
   - 页面清单
   - 功能清单
   - 字段清单
   - 规则清单
   - API 规划
   - 数据模型建议
   - 分期路线

## 3. 产品设计原则

结合当前项目现状，系统管理模块建议遵循以下原则：

### 3.1 渐进增强，不一次做重

优先完成最核心的后台管理能力：

- 用户
- 角色
- 权限
- 菜单
- 字典
- 系统参数
- 日志

组织、多租户、复杂数据权限等能力放到后续阶段。

### 3.2 保持组件化与低耦合

系统管理模块建议拆分为多个独立子模块：

- AuthModule
- UsersModule
- RolesModule
- MenusModule
- DeptsModule
- DictModule
- ConfigModule
- AuditModule

每个模块独立维护：

- entity
- dto
- controller
- service
- policy / guard / mapper

### 3.3 保持“手机号优先”的账号体系

当前项目已采用手机号作为登录标识，这个方向建议保留，不建议第一阶段强行引入用户名/邮箱体系，避免复杂度上升。

### 3.4 先满足后台管理，再考虑平台化

系统管理模块优先服务当前后台场景，避免一开始引入过度抽象（例如复杂 IAM、ABAC、多租户隔离）。

## 5. 目标用户与角色

### 5.1 用户类型

1. **超级管理员 / 平台管理员**
   - 负责系统初始化、权限分配、核心配置维护
   - 可查看全部系统管理模块

2. **普通管理员**
   - 负责用户、角色、菜单、字典、参数等日常管理
   - 权限受角色控制

3. **普通业务用户**
   - 不进入系统管理模块
   - 仅使用业务功能和个人中心

### 5.2 角色模型演进建议

#### 当前版本

- `admin`
- `user`

#### 目标版本

建议演进为：

- 角色实体表 `sys_roles`
- 用户与角色关联表 `sys_user_roles`
- 角色与权限关联表 `sys_role_permissions`

保留兼容策略：

- 第一阶段仍兼容当前 `users.role`
- 第二阶段平滑迁移到角色表

## 7. 详细功能设计

## 7.1 登录与个人安全

### 7.1.1 功能目标

提供统一的后台账号认证与安全能力，保证管理员可安全登录、刷新令牌、退出登录、修改密码。

### 7.1.2 当前已实现接口

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/refresh`
- `POST /api/auth/logout`
- `GET /api/auth/profile`
- `POST /api/auth/change-password`

### 7.1.3 建议补充能力

1. 登录失败次数限制
2. 账号禁用后禁止登录
3. 首次登录强制修改密码（可选）
4. 密码强度校验升级
5. 记录最后登录时间 / 登录 IP
6. 支持管理员重置用户密码

### 7.1.4 核心规则

1. 登录标识为手机号
2. 密码不得明文存储
3. 修改密码后，历史 refresh token 失效
4. 被禁用用户不可登录
5. 用户删除后不可继续登录

### 7.1.5 页面定义

#### 登录页
- 输入项：手机号、密码
- 操作：登录
- 状态：正常、校验失败、密码错误、账号禁用

#### 个人中心
- 展示：头像、昵称、手机号、角色、创建时间
- 操作：修改昵称、头像、密码

#### 安全设置
- 展示：最后登录时间、最后登录 IP、密码更新时间
- 操作：修改密码、退出全部设备（后续）

## 7.2 用户管理

### 7.2.1 功能目标

支持管理员完成用户全生命周期管理，包括：查询、创建、编辑、禁用、删除、分配角色、重置密码。

### 7.2.2 当前已实现能力

- 分页查询用户
- 创建用户
- 查询用户详情
- 更新用户
- 软删除用户

### 7.2.3 建议页面字段

#### 列表筛选区
- 手机号
- 昵称
- 用户状态
- 角色
- 创建时间范围

#### 列表字段
- 用户 ID
- 手机号
- 昵称
- 头像
- 角色
- 状态
- 最后登录时间
- 创建时间
- 更新时间
- 操作

#### 详情 / 编辑字段
- 手机号
- 昵称
- 头像
- 状态
- 所属部门
- 备注
- 角色（多选或单选，视权限模型阶段而定）

### 7.2.4 业务规则

1. 手机号唯一
2. 用户删除采用软删除
3. 禁用用户不可登录
4. 管理员不能删除自己（建议）
5. 管理员不能把自己降权到无系统管理权限（建议）
6. 默认注册用户角色为 `user`
7. 命中管理员白名单的用户可初始化为 `admin`

### 7.2.5 建议新增字段

在当前 `users` 基础上补充：

- `status`：启用/禁用
- `departmentId`：所属部门
- `remark`：备注
- `lastLoginAt`：最后登录时间
- `lastLoginIp`：最后登录 IP
- `passwordUpdatedAt`：密码更新时间
- `createdBy`：创建人
- `updatedBy`：更新人

### 7.2.6 建议 API

- `GET /api/users`
- `POST /api/users`
- `GET /api/users/:id`
- `PATCH /api/users/:id`
- `DELETE /api/users/:id`
- `PATCH /api/users/:id/status`
- `PATCH /api/users/:id/reset-password`
- `PATCH /api/users/:id/roles`

## 7.3 角色管理

### 7.3.1 功能目标

支持管理员定义角色、绑定权限、分配用户、控制后台功能可见与可操作范围。

### 7.3.2 当前已实现能力

- 获取角色选项
- 查询指定用户角色
- 修改指定用户角色
- 角色枚举：`admin/user`

### 7.3.3 当前局限

当前角色能力本质上还是“用户字段上的枚举”，不是完整 RBAC。

### 7.3.4 目标版本能力

#### 角色字段
- 角色名称
- 角色编码
- 角色状态
- 角色描述
- 排序
- 默认角色标记
- 数据范围（预留）

#### 角色操作
- 创建角色
- 编辑角色
- 删除角色
- 启用/停用角色
- 绑定菜单权限
- 绑定按钮权限
- 绑定 API 权限（可选）
- 查看角色下用户

### 7.3.5 业务规则

1. 系统内必须至少存在一个管理员角色
2. 被分配给用户的角色不可直接删除
3. 停用角色后，已绑定用户自动失去对应权限
4. 默认角色用于新用户初始化
5. 权限变更后，建议用户重新登录生效

### 7.3.6 建议 API

- `GET /api/roles`
- `POST /api/roles`
- `GET /api/roles/:id`
- `PATCH /api/roles/:id`
- `DELETE /api/roles/:id`
- `PATCH /api/roles/:id/status`
- `GET /api/roles/:id/permissions`
- `PATCH /api/roles/:id/permissions`
- `GET /api/roles/:id/users`

## 7.4 菜单管理

### 7.4.1 功能目标

提供后台导航结构与权限挂载能力，是 RBAC 的核心载体。

### 7.4.2 菜单类型建议

1. 目录
2. 菜单页
3. 按钮 / 操作点
4. 外链（可选）

### 7.4.3 字段建议

- 菜单名称
- 菜单编码
- 父级菜单 ID
- 类型
- 路由路径
- 前端组件路径
- 权限标识
- 图标
- 排序
- 是否缓存
- 是否隐藏
- 状态
- 备注

### 7.4.4 业务规则

1. 目录可包含子菜单，但不可直接绑定按钮权限
2. 菜单页可绑定页面权限
3. 按钮类型不直接出现在导航中
4. 角色分配菜单后，前端根据菜单树控制可见性
5. 按钮权限与后端接口权限保持命名一致

### 7.4.5 权限标识命名建议

统一采用：`模块:资源:动作`

示例：
- `system:user:view`
- `system:user:create`
- `system:user:update`
- `system:user:delete`
- `system:role:assign`
- `system:menu:update`

### 7.4.6 建议 API

- `GET /api/menus/tree`
- `GET /api/menus`
- `POST /api/menus`
- `GET /api/menus/:id`
- `PATCH /api/menus/:id`
- `DELETE /api/menus/:id`

## 7.6 字典管理

### 7.6.1 功能目标

用于统一维护业务中的枚举项、状态选项、标签映射等配置，避免魔法值散落在代码和数据库中。

### 7.6.2 设计结构

分为两层：

1. **字典类型**
   - 如：用户状态、性别、审核状态

2. **字典项**
   - 如：启用 / 禁用
   - 如：待审核 / 已通过 / 已拒绝

### 7.6.3 字段建议

#### 字典类型
- 字典名称
- 字典编码
- 状态
- 备注

#### 字典项
- 字典类型 ID
- 标签
- 值
- 标签颜色
- 排序
- 状态
- 是否默认
- 备注

### 7.6.4 业务规则

1. 字典编码全局唯一
2. 已被业务使用的字典值原则上不允许直接删除
3. 字典项可停用，但历史数据仍需兼容展示

### 7.6.5 建议 API

- `GET /api/dicts/types`
- `POST /api/dicts/types`
- `PATCH /api/dicts/types/:id`
- `DELETE /api/dicts/types/:id`
- `GET /api/dicts/items`
- `POST /api/dicts/items`
- `PATCH /api/dicts/items/:id`
- `DELETE /api/dicts/items/:id`
- `GET /api/dicts/cache/:code`

## 7.7 参数配置

### 7.7.1 功能目标

用于维护系统运行时参数，支撑后台运营配置，而不是每次都改代码或环境变量。

### 7.7.2 适合配置的内容

- 默认注册开关
- 默认头像 URL
- 登录失败锁定阈值
- 是否允许公开注册
- 默认密码策略说明
- 前端页面开关（非敏感）

### 7.7.3 不适合放入参数配置的内容

- JWT Secret
- 数据库密码
- 云厂商密钥
- 第三方凭证

上述敏感项必须继续放在环境变量或密钥系统中。

### 7.7.4 字段建议

- 参数名称
- 参数键名
- 参数值
- 值类型（string / number / boolean / json）
- 分组
- 是否系统内置
- 备注

### 7.7.5 建议 API

- `GET /api/configs`
- `POST /api/configs`
- `PATCH /api/configs/:id`
- `DELETE /api/configs/:id`
- `GET /api/configs/key/:key`

### 7.8.3 使用场景

1. 排查账号异常登录
2. 分析暴力尝试

### 7.8.4 建议 API

- `GET /api/login-logs/:id`
- `DELETE /api/login-logs/:id`（谨慎）
- `DELETE /api/login-logs/clean`（仅平台管理员）

### 7.9.1 功能目标

### 7.9.2 应记录的操作

- 新建用户
- 修改用户
- 删除用户
- 重置密码

### 7.9.3 字段建议

- 请求参数摘要

- 耗时

1. 默认不记录密码明文、token、敏感字段

### 7.9.5 建议 API

## 9. API 规范建议

### 9.1 统一规范

1. 所有系统管理接口统一走 `/api`
2. 保持当前统一响应结构：

```json
{
  "code": 200,
  "message": "请求成功",
  "data": {}
}
```

3. 保持当前统一异常结构：

```json
{
  "code": 400,
  "timestamp": "2026-03-26T02:00:00.000Z",
  "path": "/api/users",
  "message": "请求参数错误"
}
```

4. 管理端列表接口统一返回：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "pageSize": 10
}
```

### 9.2 权限校验策略

建议从“角色守卫”逐步演进为：

1. 登录鉴权：JWT
2. 角色校验：是否具备系统管理入口资格
3. 权限点校验：是否具备某一资源的操作权限

## 12. 验收标准

## 12.1 功能验收

1. 管理员可完成用户增删改查
2. 管理员可禁用/启用用户
3. 管理员可为用户分配角色
4. 角色可绑定菜单权限
5. 系统可根据角色返回可访问菜单树
6. 登录与关键管理操作可被记录
7. 字典与系统参数可在后台维护

## 12.2 安全验收

1. 被禁用用户无法登录
2. 删除用户无法继续使用 token
3. 重置密码后旧 token 失效（建议）
4. 敏感信息不进入日志
5. 权限不足用户访问管理接口时返回 403

## 12.3 体验验收

1. 列表页查询、重置、分页行为一致
2. 表单校验提示明确
3. 关键页面结构统一
4. 菜单命名、权限点命名统一

---

## 13. 风险与注意事项

1. **不要一次引入过重的 RBAC 设计**
   - 当前项目规模下，先完成菜单 + 角色 + 权限点即可

2. **不要把敏感配置搬进系统参数表**
   - Secret、数据库密码等必须留在环境变量

3. **日志记录必须脱敏**
   - 密码、token、refresh token、数据库连接串不得进入日志

4. **优先兼容当前代码结构**
   - 当前已有 `users`、`auth`、`roles`，建议在此基础上平滑演进，而不是推倒重做

---

## 14. 推荐开发顺序

建议按下面顺序推进，最稳：

1. 用户状态 + 重置密码
2. 登录日志 + 操作日志
3. 菜单管理 + 权限标识体系
4. 角色实体化
5. 前端菜单树与按钮权限联动
6. 部门管理
7. 字典管理
8. 参数配置

---

## 15. 结论

`nest-admin` 当前已经具备“系统管理模块”的基础骨架，但仍停留在**认证 + 用户 + 简化角色**阶段。

后续最合理的演进方向不是盲目做大，而是围绕以下主线补齐：

- 用户闭环
- 角色闭环
- 菜单闭环
- 日志闭环
- 配置闭环

这样既能保持项目轻量，又能让系统管理模块真正达到“可交付、可扩展、可维护”的后台标准。
