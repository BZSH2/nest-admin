# Nest Admin 现有产品文档

## 1. 文档说明

- 文档名称：Nest Admin 现有产品文档
- 文档口径：**仅描述当前代码已经实现并可交付的能力**
- 文档目的：用于产品、前端、测试、运维快速理解当前 `nest-admin` 的实际功能边界
- 适用版本：当前仓库现状（基于 `auth / users / roles / openapi / health` 模块）

---

## 2. 产品概述

### 2.1 产品定位

`Nest Admin` 当前是一个**后台管理系统后端基础服务**，主要提供：

1. 用户认证
2. 用户资料管理
3. 管理员用户管理
4. 基础角色管理
5. 健康检查
6. OpenAPI / Swagger 文档输出能力

当前产品形态更接近：

- **后台管理系统后端骨架**
- **轻量 RBAC 骨架服务**
- **适合继续扩展系统管理模块的基础工程**

### 2.2 当前已落地模块

当前已实现的模块有：

- 应用基础模块（根路由 / 健康检查）
- 认证模块 `Auth`
- 用户模块 `Users`
- 角色模块 `Roles`
- OpenAPI 模块 `OpenAPI`

### 2.3 当前目标用户

当前系统主要面向两类用户：

1. **管理员（admin）**
   - 可访问用户管理接口
   - 可访问角色管理接口
   - 具备后台系统管理权限

2. **普通用户（user）**
   - 可注册、登录、查看个人资料、修改密码
   - 不可访问用户管理和角色管理接口

## 4. 当前产品架构

## 4.1 系统角色模型

当前角色是**轻量枚举模型**，不是完整角色实体系统。

当前仅支持两个角色：

- `admin`：管理员
- `user`：普通用户

### 4.1.1 角色来源规则

当前角色来源分两层：

1. **数据库中的 `users.role` 字段**
2. **管理员手机号白名单回退识别**

即：
- 如果用户在数据库里已有角色，以数据库角色为准
- 如果数据库角色为空，则可通过环境变量中的管理员手机号白名单识别是否为管理员

### 4.1.2 管理员白名单

当前系统支持通过环境变量初始化管理员手机号：

```env
ADMIN_PHONE_NUMBERS=13800138000,13900139000
```

命中白名单的用户可自动识别为管理员角色。

## 5. 当前用户体系

## 5.1 当前用户模型

当前 `users` 表中的核心字段包括：

- `id`：用户 ID（UUID）
- `phoneNumber`：手机号，唯一
- `password`：加密密码
- `role`：用户角色，可为空
- `nickname`：昵称
- `avatar`：头像 URL
- `currentHashedRefreshToken`：当前 refresh token 哈希
- `createdAt`：创建时间
- `updatedAt`：更新时间
- `deletedAt`：软删除时间

## 5.2 当前用户特征

当前用户模型是**轻量账号模型**，已具备以下特性：

1. 手机号作为唯一登录标识
2. 密码加密存储
3. 支持 refresh token 机制
4. 支持软删除
5. 支持基础角色识别
6. 支持昵称与头像资料

## 5.3 当前未具备的用户能力

当前系统**尚未实现**以下常见后台用户能力：

- 用户状态（启用/禁用）
- 部门归属
- 最后登录时间
- 最后登录 IP
- 备注字段
- 管理员重置密码
- 多角色用户

## 6. 当前接口与产品说明

## 6.1 公共与系统基础接口

### 6.1.1 根接口
**接口：** `GET /`

**用途：**
- 用于服务基础探活
- 当前返回固定文案：`Hello World!`

### 6.1.2 健康检查
当前已提供 3 个健康接口：

- `GET /health`
- `GET /health/live`
- `GET /health/ready`

#### 存活检查
- `/health`
- `/health/live`

返回系统基础运行状态，包括：
- `status`
- `timestamp`
- `uptime`

#### 就绪检查
- `/health/ready`

在基础状态上额外校验数据库是否可用：
- 若数据库已初始化且可连接，返回 `database: up`
- 若数据库未就绪，则返回服务不可用错误

### 6.1.3 Swagger 文档
**访问地址：** `/docs`

**用途：**
- 提供当前接口文档
- 用于前后端联调
- 可查看请求参数、响应模型、权限要求

## 6.2 认证模块（Auth）

## 6.2.1 模块定位
认证模块负责用户账号的登录态建立与个人安全能力，是当前系统的核心入口模块。

## 6.2.2 注册
**接口：** `POST /auth/register`

### 功能说明
允许用户通过手机号完成注册。

### 请求字段
- `phoneNumber`：手机号，11 位，必填
- `password`：密码，6-20 位，必填
- `nickname`：昵称，2-20 位，可选

### 当前规则
1. 手机号必须符合大陆手机号格式
2. 手机号不可重复
3. 密码会加密保存
4. 新注册用户默认会写入基础角色
5. 若手机号命中管理员白名单，可初始化为管理员角色

### 返回结果
返回当前注册用户资料，不直接返回 token。

返回资料包含：
- 用户 ID
- 手机号
- 昵称
- 头像
- 角色
- 创建时间
- 更新时间

## 6.2.3 登录
**接口：** `POST /auth/login`

### 功能说明
用户通过手机号和密码登录系统，成功后获取 token 对。

### 请求字段
- `phoneNumber`
- `password`

### 当前规则
1. 手机号和密码必须匹配
2. 密码校验失败返回未授权
3. 登录成功后生成：
   - `accessToken`
   - `refreshToken`
4. 系统会保存当前 refresh token 的哈希值，用于后续刷新校验

### 返回结果
- `accessToken`
- `refreshToken`

## 6.2.4 登出
**接口：** `POST /auth/logout`

### 功能说明
当前登录用户主动退出系统。

### 当前规则
1. 需要登录态
2. 登出后会清除当前用户保存的 refresh token 哈希
3. 返回操作结果消息

## 6.2.5 刷新 Token
**接口：** `POST /auth/refresh`

### 功能说明
使用 refresh token 换取新的 token 对。

### 当前规则
1. 必须携带合法 refresh token
2. refresh token 必须与数据库中当前保存的哈希匹配
3. 刷新成功后会重新生成新的 token 对
4. 同时会更新当前 refresh token 哈希

## 6.2.6 获取个人资料
**接口：** `GET /auth/profile`

### 功能说明
获取当前登录用户资料。

### 返回字段
- `id`
- `phoneNumber`
- `nickname`
- `avatar`
- `role`
- `createdAt`
- `updatedAt`

## 6.2.7 修改密码
**接口：** `POST /auth/change-password`

### 功能说明
当前登录用户修改自己的密码。

### 请求字段
- `oldPassword`：旧密码
- `newPassword`：新密码（6-20 位）

### 当前规则
1. 需要登录态
2. 旧密码必须正确
3. 新密码会重新加密保存
4. 修改成功后当前 refresh token 会被移除
5. 用户需要重新登录

## 6.3 用户管理模块（Users）

## 6.3.1 模块定位
用户管理模块提供后台用户的基础管理能力。

### 访问权限
当前用户管理接口默认要求：
- 已登录
- 角色为 `admin`

## 6.3.2 分页查询用户
**接口：** `GET /users`

### 功能说明
分页查询用户列表。

### 支持查询参数
- `page`：页码，默认 1
- `pageSize`：每页数量，默认 10
- `keyword`：关键字，支持按手机号 / 昵称模糊搜索

### 返回结构
- `items`
- `total`
- `page`
- `pageSize`

### 当前列表字段
返回的用户字段包含：
- `id`
- `phoneNumber`
- `role`
- `nickname`
- `avatar`
- `createdAt`
- `updatedAt`

## 6.3.3 创建用户
**接口：** `POST /users`

### 功能说明
管理员手动创建用户。

### 请求字段
- `phoneNumber`
- `password`
- `nickname`

### 当前规则
1. 手机号唯一
2. 密码加密保存
3. 新用户角色按当前角色识别规则初始化
4. 返回安全用户信息，不返回密码

## 6.3.4 查询单个用户
**接口：** `GET /users/:id`

### 功能说明
管理员根据用户 ID 查看指定用户信息。

### 当前规则
1. 用户不存在时返回 404
2. 只返回安全字段，不返回密码与 refresh token

## 6.3.5 更新用户
**接口：** `PATCH /users/:id`

### 功能说明
管理员更新指定用户信息。

### 当前可更新字段
- `phoneNumber`
- `password`
- `nickname`

### 当前规则
1. 若修改手机号，仍需保证唯一
2. 若更新密码，会重新进行加密
3. 返回更新后的安全用户信息

## 6.3.6 删除用户
**接口：** `DELETE /users/:id`

### 功能说明
管理员删除用户。

### 当前规则
1. 采用**软删除**
2. 用户不存在时返回 404
3. 删除成功后返回被删除用户 ID

## 6.4 角色管理模块（Roles）

## 6.4.1 模块定位
角色管理模块当前提供的是**基础角色分配能力**，用于支撑系统管理权限入口。

### 访问权限
当前角色管理接口默认要求：
- 已登录
- 角色为 `admin`

## 6.4.2 获取角色选项
**接口：** `GET /roles/options`

### 功能说明
返回当前系统中可分配角色列表。

### 当前返回值
目前固定返回两个角色：
- `admin` / 管理员
- `user` / 普通用户

## 6.4.3 获取指定用户角色
**接口：** `GET /roles/users/:userId`

### 功能说明
查看指定用户当前角色信息。

### 返回字段
- `userId`
- `phoneNumber`
- `nickname`
- `avatar`
- `role`

## 6.4.4 更新指定用户角色
**接口：** `PATCH /roles/users/:userId`

### 功能说明
管理员修改指定用户角色。

### 请求字段
- `role`

### 当前可选值
- `admin`
- `user`

### 当前规则
1. 仅支持单角色赋值
2. 用户不存在返回 404
3. 角色值不合法时返回参数错误
4. 更新成功后返回新的角色详情信息

## 6.5 OpenAPI 模块

## 6.5.1 模块定位
该模块用于输出系统 OpenAPI 信息，便于接口聚合、模块化文档分发和自动化集成。

## 6.5.2 当前接口

### 获取模块列表
- `GET /getModules`

返回当前可用的 Swagger 模块与服务定义。

### 按模块获取 OpenAPI JSON
- `POST /postOpenApiJson`

根据模块筛选生成 OpenAPI 文档 JSON。

### 获取 OpenAPI 接口定义
- `GET /getPostOpenApiJsonDefinition`

返回 `postOpenApiJson` 本身的 OpenAPI 定义。

## 6.5.3 当前价值

该能力适合：
- 对接外部工具
- 模块化接口文档输出
- 自动生成客户端或中台能力接入文档

## 7. 当前权限与访问控制

## 7.1 当前鉴权方式

当前系统使用：
- JWT 登录鉴权
- Refresh Token 刷新机制

## 7.2 当前权限控制方式

当前权限控制是**轻量 RBAC 骨架**：

1. 公共接口通过 `Public` 标记开放
2. 非公共接口默认要求登录
3. 管理型接口通过 `Roles(admin)` 约束
4. 角色守卫统一处理管理员权限校验

## 7.3 当前开放接口

当前公开可访问接口包括：

- `GET /`
- `GET /health`
- `GET /health/live`
- `GET /health/ready`
- `POST /auth/register`
- `POST /auth/login`
- `GET /getModules`
- `POST /postOpenApiJson`
- `GET /getPostOpenApiJsonDefinition`
- Swagger 文档入口 `/docs`

## 7.4 当前受保护接口

需要登录：
- `/auth/logout`
- `/auth/refresh`（需 refresh token）
- `/auth/profile`
- `/auth/change-password`

需要管理员权限：
- `/users/*`
- `/roles/*`

## 8. 当前响应与异常规范

## 8.1 成功响应结构

当前系统统一成功响应格式：

```json
{
  "code": 200,
  "message": "请求成功",
  "data": {}
}
```

## 8.2 异常响应结构

当前系统统一异常响应格式：

```json
{
  "code": 400,
  "timestamp": "2026-03-26T02:00:00.000Z",
  "path": "/api/auth/login",
  "message": "手机号或密码错误"
}
```

## 8.3 当前异常提示特点

当前系统已经对常见错误信息做了中文化处理，包括：

- 参数错误
- 未授权
- 禁止访问
- 资源不存在
- 冲突
- 服务不可用

这意味着当前系统在前端联调时，接口返回已经具备一定可直接展示性。

## 11. 当前版本结论

从产品视角看，`nest-admin` 当前已经不是一个空壳项目，而是一个具备**基础可用性**的后台后端系统，核心价值体现在：

1. 已经跑通了完整的账号闭环
2. 已经具备管理员管理用户的能力
3. 已经具备最基础的角色分配与权限入口控制
4. 已经具备工程化需要的健康检查与文档输出

但它当前仍然属于：

> **系统管理模块的基础版 / 骨架版**

也就是说，当前版本适合：
- 用起来
- 联调起来
- 继续扩起来

但还不能算“完整后台系统管理平台”。
